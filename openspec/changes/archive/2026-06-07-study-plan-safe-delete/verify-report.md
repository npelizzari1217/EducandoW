## Findings

### WARNING — Message built in domain layer, not presentation layer

**File**: `packages/domain/src/pedagogy/errors/study-plan.errors.ts`
**Spec reference**: Domain Invariants: "Message templates live in presentation layer only"

The tasks file explicitly noted "(orchestrator override: message built in constructor)". The Spanish strings are generated by `buildMessage()` inside `StudyPlanHasDependenciesError`, meaning the domain error carries a fully formatted human-readable message. The controller reads `e.message` directly.

**Impact**: All 6 message strings match the spec exactly; behavior is correct. Risk is limited to future i18n or message customization — at that point the coupling must be unwound. Currently zero functional risk.

**Resolution**: Intentional deviation confirmed by orchestrator override. No code change needed; document the decision.

---

### SUGGESTION — AlertModal lacks aria role/modal attributes

**File**: `web/src/components/ui/alert-modal.tsx:14`

The overlay div has no `role="dialog"`, `aria-modal="true"`, or `aria-labelledby`. Screen readers will not announce this as a dialog.

**Resolution**: Add `role="dialog" aria-modal="true" aria-labelledby="alert-modal-title"` and an `id="alert-modal-title"` on the `<h2>`.
