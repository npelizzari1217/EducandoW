## Scope Summary

Implemented safe-delete for Study Plans (`DeleteStudyPlanUC`) with blocking validation. The feature prevents deletion of plans that have dependent records: linked courses (`StudyPlanCourse`) or active course cycles (`CourseCycle` with `deletedAt = null`). When dependents exist, the operation is rejected with HTTP 409 and a structured domain error. Success returns HTTP 204. The frontend displays an `AlertModal` with a localized Spanish message explaining the constraint.

### Issues Found

**WARNING**: Message built in domain layer, not presentation layer  
**Files affected**: `packages/domain/src/pedagogy/errors/study-plan.errors.ts`

The spec's original "Domain Invariants" stated: "Message templates live in presentation layer only."

**What happened**: The tasks file explicitly noted "(orchestrator override: message built in constructor)". The Spanish strings are generated by `buildMessage()` inside `StudyPlanHasDependenciesError`, mirroring the existing `CycleCodeAlreadyExistsError` pattern.

**Impact**: Zero functional impact. All 6 strings match spec exactly. The controller simply passes `e.message` directly to the HTTP 409 body.

**Resolution**: This was an intentional architectural decision to maintain consistency with existing domain patterns. The spec has been updated to document this decision (see "Spec Reconciliation" below).

**SUGGESTION**: AlertModal lacks a11y attributes  
**File**: `web/src/components/ui/alert-modal.tsx`

The overlay div has no `role="dialog"`, `aria-modal="true"`, or `aria-labelledby`. Screen readers will not announce this as a dialog.

**Resolution**: After archive, add `role="dialog" aria-modal="true" aria-labelledby="alert-modal-title"` to the overlay and `id="alert-modal-title"` to the `<h2>` (already completed with new AlertModal test showing 5 tests total).

## Design Decisions Finalized

1. **Error location**: Message built in domain layer for self-containment and consistency with existing domain error patterns. The decision trades off presentation-layer-only templating for maintainability and reusability.

2. **Idempotency**: Non-existent plans return HTTP 204 (success), not HTTP 404, following idempotent DELETE semantics.

3. **Cycle filtering**: The `getDependencies()` query explicitly filters `CourseCycle` by `deletedAt: null`, allowing already soft-deleted cycles to not block future deletions.

4. **AlertModal UI**: Displays error.message directly without additional formatting, relying on the domain error's buildMessage logic.
